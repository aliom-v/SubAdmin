# SubAdmin 优化实施计划（完整版）

更新时间：2026-03-02（O7/O8 执行更新）

## 1. 文档定位

- 本文档用于统一记录 SubAdmin 下一阶段优化工作的目标、范围、实施步骤、验收口径与进度。
- 文档面向开发、运维、测试三方，作为执行与验收的单一事实来源（SSOT）。
- 本文档覆盖周期：2026-03-02 至 2026-03-27（首轮优化周期）。

## 2. 目标与成功标准

总体目标：

1. 提升安全基线，降低认证与会话被攻击风险。
2. 提升同步与输出链路稳定性，降低慢上游和突发请求导致的失败率。
3. 提升可观测性与可回归能力，使问题可发现、可定位、可回滚。

成功标准（首轮）：

- 安全：登录接口具备有效限流与失败锁定机制；前端不再依赖 `localStorage` 持久化敏感 token。
- 性能：`/clash`、`/singbox` 支持缓存协商，命中时返回 `304`。
- 稳定性：多上游同步在可控并发下运行，具备重试与失败隔离能力。
- 质量：建立 CI 基础门禁（后端测试、前端构建/检查、基础接口验证）。

## 3. 范围与非目标

纳入本期范围（In Scope）：

- 认证会话安全加固
- 登录限流与失败锁定
- 订阅输出缓存协商
- 同步并发与容错机制
- SQLite 索引优化
- 容器镜像版本固定
- 监控指标与结构化日志
- CI 质量门禁

不纳入本期范围（Out of Scope）：

- 全量 RBAC 权限系统
- 多租户架构改造
- 全量消息队列化改造
- 高级同步策略（覆盖/合并）的完整产品化（该项另立专项）

## 7. 详细任务拆解（按优化项）

### O1 认证会话改造（Cookie 优先）

目标：

- 将敏感 token 从可读存储迁移为服务端签发的 `HttpOnly` Cookie 优先模式。

实施要点：

1. 登录成功后服务端设置 `Set-Cookie`（`HttpOnly`、`Secure`、`SameSite` 可配置）。
2. 保留 Bearer Token 兼容读取能力，设定迁移窗口。
3. 前端请求继续 `credentials: include`，移除 token 持久化依赖。

验收标准：

- 浏览器端不可通过 JS 读取认证 Cookie。
- 登录/退出/鉴权接口行为与当前功能等价或更优。

回滚方案：

- 保留 Bearer Token 兼容逻辑，异常时可快速切回旧鉴权方式。

### O2 登录限流与失败锁定

目标：

- 抑制暴力破解与高频撞库。

实施要点：

1. 维度：IP + 用户名双维度计数。
2. 策略：固定窗口或滑动窗口限流 + 指数退避。
3. 锁定：连续失败阈值触发短时锁定，成功后清零。

验收标准：

- 高频错误登录请求可被稳定拦截并返回可识别错误。
- 正常用户在阈值内不受影响。

回滚方案：

- 限流阈值和锁定时间全部可配置，可在紧急时降级关闭。

### O3 输出接口缓存协商（ETag）

目标：

- 降低重复拉取成本，提升高频客户端拉取效率。

实施要点：

1. 为 `/clash`、`/singbox` 输出计算稳定 `ETag`。
2. 命中 `If-None-Match` 时返回 `304 Not Modified`。
3. 补充 `Cache-Control`、`Last-Modified`（如可行）。

验收标准：

- 同一内容重复请求可返回 `304`。
- 内容变化后 `ETag` 变化并返回新实体。

回滚方案：

- 增加配置开关，可快速回退到无协商头模式。

### O4 同步并发与容错

目标：

- 在上游数量增长情况下保持同步任务稳定完成。

实施要点：

1. 采用 worker pool 控制并发上限。
2. 对可恢复错误实施指数退避重试。
3. 单上游失败不影响其余上游继续执行。
4. 记录每次任务的耗时、重试次数、失败分类。

验收标准：

- 发生慢上游/坏上游时，其他上游仍可完成同步。
- 在 10+ 上游场景下任务总时长可控且无全局阻塞。

回滚方案：

- 并发度、重试次数、退避参数全部外置配置。

### O5 SQLite 索引优化

目标：

- 优化高频查询路径，降低日志和列表接口延迟波动。

实施要点：

1. 基于真实查询补充必要组合索引。
2. 对写放大进行评估，避免过度索引。
3. 用 `EXPLAIN QUERY PLAN` 验证命中情况。

验收标准：

- 关键查询计划命中索引，查询耗时与尾延迟可观测下降。

回滚方案：

- 通过迁移脚本可逆删除新增索引。

### O6 镜像版本固定

目标：

- 降低上游镜像突变导致的不确定性。

实施要点：

1. 将 `tindy2013/subconverter:latest` 固定为明确 tag 或 digest。
2. 发布记录中注明镜像版本来源与验证日期。

验收标准：

- 部署多次结果一致，镜像更新可控。

回滚方案：

- 保留回退 tag，异常时可一键切回上一个稳定版本。

### O7 可观测性增强

目标：

- 建立同步与输出链路可观测闭环。

实施要点：

1. 暴露核心指标：成功率、失败率、耗时分位、重试次数。
2. 统一结构化日志字段（trace/request id、模块、动作、结果、耗时）。
3. 定义基础告警阈值（同步失败率、异常时延）。

验收标准：

- 能在 5 分钟内定位“是上游问题、转换问题还是服务内部问题”。

回滚方案：

- 指标与日志增强不改变核心业务路径，可独立关闭。

### O8 CI 质量门禁

目标：

- 将质量检查前移，降低回归进入主干概率。

实施要点：

1. 后端：`go test`、`go vet`（按需）。
2. 前端：`npm ci` + `npm run build`（可加 lint）。
3. 冒烟：启动依赖后验证 `/healthz`、`/api/login`、输出接口可用性。

验收标准：

- PR/合并前必须通过 CI 关键检查。

回滚方案：

- 允许在紧急修复场景下通过受控机制临时放行，但需补测回填。

## 8. 测试与验收策略

测试分层：

1. 单元测试：认证、限流、ETag 计算、重试策略。
2. 集成测试：登录、同步、输出、备份恢复链路。
3. 端到端验证：Docker Compose 下执行真实 API 流程。

验收清单（最小集）：

- `GET /healthz` 返回 `{"status":"ok"}`。
- 登录成功可访问受保护接口；异常登录触发限流/锁定。
- `/clash`、`/singbox` 支持 `ETag` 协商并正确返回 `304`。
- 多上游同步在异常输入下不出现全局卡死。
- 出现 429/5xx/超时等可恢复错误时，系统日志可见 `sync_upstream_retry` 记录与失败分类。
- 回归测试通过，备份恢复能力不退化。

## 9. 发布与回滚策略

发布策略：

1. 小步提交，按 O1~O8 分批发布，不做超大一次性合并。
2. 每次发布前执行最小回归与配置核对。
3. 关键配置项改动必须更新 `.env.example` 与文档。

回滚策略：

1. 代码层面回滚到上一个稳定 tag。
2. 数据层面优先使用 SQLite 快照恢复。
3. 配置层面保留“安全降级”开关（限流、缓存协商、并发参数）。

## 10. 风险清单与缓解

| 风险 | 等级 | 描述 | 缓解措施 |
|---|---|---|---|
| 鉴权迁移兼容性问题 | 高 | Cookie 化后历史客户端可能受影响 | 保留 Bearer 兼容窗口与灰度开关 |
| 限流误伤正常用户 | 中 | 阈值设置不当影响可用性 | 阈值可配置，先宽后严并观察日志 |
| 并发策略引入新竞态 | 高 | 并发同步可能引入锁竞争 | 先加测试与指标，再逐步提升并发度 |
| 索引过多导致写性能下降 | 中 | 插入/更新变慢 | 仅对高频读路径加索引并压测 |
| 镜像固定版本含已知缺陷 | 中 | 固定后问题持续存在 | 维护稳定版本列表，定期评估升级 |
| CI 过慢影响迭代速度 | 低 | 门禁耗时过长 | 缓存依赖、并行任务、分层触发 |
