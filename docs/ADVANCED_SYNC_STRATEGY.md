# 高级同步策略设计与实现状态

更新时间：2026-03-08

## 1. 文档定位

- 本文档用于定义 SubAdmin“高级同步策略”的需求、规则、API、验收标准与实施拆分，并同步当前实现状态。
- 当前仓库已完成 V1 最小闭环：后端策略 API、聚合链路接入、管理后台配置区与 preview 结果展示。
- 本文档仍保留后续产品化方向，避免把“已实现”和“拟设计”混在代码或 README 中。

## 2. 需求

### 2.1 背景

当前系统已经支持：

- 多上游订阅与手动节点统一汇总
- 同步并发、重试、失败隔离
- 固定输出 `/clash`、`/singbox`
- 缓存模式、ETag/304、备份恢复、快照回滚

当前仍未完全产品化的能力在于：

- 多来源节点在“同名、重复、冲突”时如何处理
- 手动节点与上游节点谁优先
- 不同输出目标是否允许使用不同聚合策略
- 管理员如何在执行前预览策略结果，而不是只能依赖最终输出

这也是 `docs/STATUS.md` 中仍标注为“进行中”的模块。

### 2.2 目标

V1 目标：

1. 让管理员可以明确控制“覆盖 / 合并 / 冲突处理”行为。
2. 让不同来源的优先级可配置，避免结果不透明。
3. 在真正写入输出前，支持预览策略执行结果与差异摘要。
4. 保持与当前同步、缓存、日志、备份能力兼容。

### 2.3 非目标

V1 不包含：

- 完整 RBAC 下的多角色策略审批流
- 针对每个节点做复杂脚本化规则引擎
- 按地区、协议、标签进行高级路由编排
- 为 Clash / Sing-box 分别维护完全独立的策略 DSL
- 自动智能识别“最佳节点”并实时动态切换

### 2.4 典型场景

场景 A：多个上游都包含相同节点

- 期望：自动去重，不因来源重复而重复出现在输出中。

场景 B：不同上游提供同名但内容不同的节点

- 期望：按管理员定义的优先级保留其一，或重命名后同时保留。

场景 C：手动节点需要覆盖上游节点

- 期望：手动节点可配置为最高优先级，在冲突时优先保留。

场景 D：`/clash` 与 `/singbox` 想采用相同聚合结果

- 期望：V1 默认共用同一套聚合策略，避免配置复杂度失控。

场景 E：改策略前需要先看影响

- 期望：支持 dry-run/preview，展示“新增、删除、重命名、冲突解决”摘要。

## 3. 规则

### 3.1 V1 范围内的策略对象

V1 采用“全局一套聚合策略 + 来源优先级”的最小模型：

- 全局策略：控制系统默认的合并与冲突处理方式
- 来源优先级：控制手动节点与各上游在冲突时的胜负顺序
- 预览能力：在不落库、不刷新输出的前提下查看执行结果

V1 暂不支持：

- 为 `clash` / `singbox` 单独配置不同同步策略
- 为单个上游设置完全不同的冲突规则
- 为单个节点打例外规则

### 3.2 策略字段建议

当前已实现或保留的配置概念：

- `strategy_mode`
  - `merge_dedupe`：合并并去重（默认）
  - `priority_override`：按优先级覆盖冲突节点
  - `keep_both_rename`：冲突时两者保留，但低优先级项自动重命名
- `manual_nodes_priority`
  - 整数，默认高于普通上游
- `upstream_priority`
  - 每个上游单独配置整数，数值越小优先级越高
- `rename_suffix_format`
  - 用于 `keep_both_rename`，默认追加来源标识，如 `节点A [upstream-1]`
- `dedupe_scope`
  - 当前仍未开放到实现层；V1 现阶段固定按节点内容去重

### 3.3 聚合执行顺序

建议固定执行顺序如下：

1. 读取所有启用来源（上游 + 手动节点）
2. 解析为标准节点列表
3. 清洗空值、非法项与不可识别项
4. 基于标准化内容做重复检测
5. 对“同名不同内容”冲突按策略模式处理
6. 生成最终聚合结果
7. 若为 preview，仅返回结果摘要；若为 sync，继续刷新输出缓存

### 3.4 冲突定义

V1 需要区分两类问题：

- 重复：标准化后节点内容一致，只是来自不同来源
- 冲突：显示名相同或语义相近，但节点内容不同

V1 推荐定义：

- 重复节点：保留最高优先级来源的一条
- 冲突节点：按 `strategy_mode` 决定覆盖或保留并重命名

### 3.5 各模式规则

#### `merge_dedupe`

- 目标：最大限度合并来源，同时消除明显重复
- 规则：
  - 内容完全相同：去重，仅保留优先级最高来源
  - 名称相同但内容不同：默认同时保留，并按来源补后缀
- 适用：默认推荐模式，风险最低

#### `priority_override`

- 目标：让高优先级来源在冲突时稳定胜出
- 规则：
  - 内容完全相同：去重，仅保留优先级最高来源
  - 名称相同但内容不同：仅保留优先级最高来源
  - 手动节点可作为最高优先级来源
- 适用：管理员希望输出稳定、命名唯一、来源有主次时

#### `keep_both_rename`

- 目标：最大化保留节点来源信息
- 规则：
  - 内容完全相同：仍去重
  - 名称相同但内容不同：全部保留，低优先级项自动重命名
- 适用：需要人工观察不同来源质量差异时

### 3.6 失败与回退规则

- 单个上游同步失败时，继续沿用已有失败隔离逻辑
- 策略计算失败时，不应覆写现有缓存输出
- preview 失败时，仅返回错误信息，不产生副作用
- 若策略配置非法，保存接口必须拒绝并返回明确错误

### 3.7 日志与可观测性

当前已实现/建议保留的结构化日志字段：

- `module=strategy`
- `action=preview_strategy` / `update_strategy`
- `strategy_mode`
- `source_count`
- `input_nodes`
- `deduped_nodes`
- `renamed_nodes`
- `dropped_nodes`
- `duration_ms`

当前已接入的策略指标：

- `subadmin_strategy_preview_total`
- `subadmin_strategy_apply_total`
- `subadmin_strategy_conflicts_total`
- `subadmin_strategy_dropped_nodes_total`

## 4. API

### 4.1 设计原则

- 尽量复用当前 `/api/settings`、`/api/upstreams`、`/api/sync` 的心智模型
- preview 与 apply 明确分离
- 写操作必须可审计，便于后续排障与回滚

### 4.2 建议接口

#### 查询当前策略

- `GET /api/strategy`

返回建议：

```json
{
  "strategy_mode": "merge_dedupe",
  "manual_nodes_priority": 0,
  "rename_suffix_format": "[{source}]",
  "upstreams": [
    { "id": 1, "name": "机场 A", "priority": 10 },
    { "id": 2, "name": "机场 B", "priority": 20 }
  ]
}
```

#### 更新策略

- `PUT /api/strategy`

说明：

- 仅更新策略配置，不直接触发同步
- 保存后由管理员决定是否执行 preview 或 apply
- 当前实现中，若系统处于缓存模式，保存后会自动刷新输出缓存

#### 预览策略结果

- `POST /api/strategy/preview`

请求建议：

```json
{
  "strategy_mode": "priority_override",
  "manual_nodes_priority": 0,
  "rename_suffix_format": "[{source}]",
  "upstreams": [
    { "id": 1, "priority": 10 },
    { "id": 2, "priority": 20 }
  ]
}
```

返回建议：

```json
{
  "summary": {
    "source_count": 3,
    "input_nodes": 120,
    "output_nodes": 86,
    "deduped_nodes": 22,
    "renamed_nodes": 8,
    "dropped_nodes": 4,
    "conflict_groups": 6
  },
  "conflicts": [
    {
      "name": "HK-01",
      "resolution": "kept_higher_priority",
      "winner_source": "机场 A",
      "dropped_sources": ["机场 B"]
    }
  ],
  "preview_nodes": [
    "HK-01",
    "JP-02",
    "US-03 [机场 B]"
  ]
}
```

#### 应用策略并触发同步

V1 建议沿用两步式流程，不额外提供“一键 apply + sync”接口：

1. `PUT /api/strategy`
2. `POST /api/sync`

理由：

- 与现有系统心智一致
- 更容易控制变更时机
- 更便于日志审计与回滚

### 4.3 数据存储建议

V1 建议两层存储：

- 全局策略主配置：放入 `settings`
- 上游优先级：优先考虑为 `upstreams` 表加字段 `priority`

如后续策略维度增多，再考虑独立表，例如：

- `sync_strategies`
- `sync_strategy_upstreams`

### 4.4 向后兼容

- 老版本未配置策略时，默认视为 `merge_dedupe`
- 现有 `/api/sync`、`/clash`、`/singbox` 行为不应因未配置策略而改变
- preview 属于新增能力，不影响当前调用链

## 5. 验收

### 5.1 功能验收

1. 未配置策略时，系统采用默认 `merge_dedupe`。
2. 两个上游内容相同的节点不会重复输出。
3. 两个上游同名不同内容节点在 `priority_override` 下仅保留高优先级项。
4. 两个上游同名不同内容节点在 `keep_both_rename` 下均保留，且名称可区分。
5. 手动节点在设置更高优先级后，可覆盖同名上游节点。
6. `POST /api/strategy/preview` 不修改缓存、不触发输出变更。
7. 策略保存后执行 `POST /api/sync`，最终输出符合预览摘要。
8. 非法策略参数会被拒绝保存，并返回明确错误。

### 5.2 回归验收

1. `./scripts/phase4_acceptance.sh` 现有能力不被破坏。
2. `/healthz`、`/metrics`、登录、备份、快照回滚保持可用。
3. 输出缓存模式与实时模式都可正常工作。
4. 同步失败隔离、重试与日志仍符合当前设计。

### 5.3 观测验收

1. 策略 preview 与 apply 会产生结构化日志。
2. 指标可观测到 preview 次数、冲突数、丢弃数。
3. 当策略导致输出节点大幅下降时，日志中可定位原因。
