# 项目遗留问题与后续优化清单

更新时间：2026-03-08

## 1. 文档定位

- 本文档用于集中记录当前项目仍未闭环的问题、文档缺口、验证缺口与后续优化建议。
- 它不替代 `docs/STATUS.md` 的状态总览，而是补充“下一步应该做什么、为什么做、优先级如何排”。
- 目标是让后续开发不必从 README、状态文档和历史对话中重新拼接上下文。

## 2. 当前判断

- 项目主线功能已经可用，核心 MVP 不处于“不可交付”状态。
- 高级同步策略已经完成最小闭环：后端 API、输出聚合接入、管理后台配置区、preview 展示与基础回归脚本断言均已落地。
- 当前主要问题已经从“缺功能”转为“缺完整验证、缺指标、缺进一步产品化能力”。
- 因此，后续工作应优先围绕“验证闭环”和“可维护性”推进，而不是继续无边界扩展新功能。

## 3. 已完成但仍需收尾的事项

### 3.1 高级同步策略

当前已完成：

- `GET /api/strategy`
- `PUT /api/strategy`
- `POST /api/strategy/preview`
- 聚合链路接入：`/clash`、`/singbox` 已使用策略计算结果
- 管理后台已支持策略模式、手动节点优先级、上游优先级与 preview 展示
- `scripts/phase4_acceptance.sh` 已补充策略相关断言

当前未完全闭环：

- 还没有策略相关指标（preview 次数、冲突数、丢弃数）
- 还没有接口级集成测试（当前以纯函数单测和回归脚本为主）

### 3.2 文档状态

当前已对齐：

- `README.md`
- `docs/STATUS.md`
- `docs/ADVANCED_SYNC_STRATEGY.md`
- `docs/STRATEGY_ROLLOUT.md`
- `docs/PHASE4_REGRESSION.md`

当前仍建议补充：

## 4. 仍然存在的问题

### P0：必须优先处理

#### 4.1 策略相关可观测性仍不完整

问题：

- 当前已有策略日志，但还没有独立指标。

影响：

- 只能通过日志排障，无法直接在监控层看到 preview 次数、冲突数量、丢弃节点数量。

建议动作：

- 在 `/metrics` 中补充：
  - `subadmin_strategy_preview_total`
  - `subadmin_strategy_apply_total`
  - `subadmin_strategy_conflicts_total`
  - `subadmin_strategy_dropped_nodes_total`

### P1：高价值优化

#### 4.2 后端缺少策略接口级测试

问题：

- 当前已经有 `strategy` 纯函数单测，但还缺少 HTTP 层和持久化层的集成测试。

影响：

- API 请求/响应、配置持久化和缓存模式下自动刷新行为，仍主要依赖人工验证和回归脚本。

建议动作：

- 补充后端接口测试，至少覆盖：
  - 默认策略读取
  - 非法 `strategy_mode` 拒绝
  - `PUT /api/strategy` 保存后再次读取
  - 缓存模式下保存策略后自动刷新缓存

#### 4.3 前端主文件过大，维护成本升高

问题：

- `web/src/App.jsx` 已承担登录、上游、节点、设置、策略、备份、日志等全部视图逻辑。

影响：

- 后续再扩展策略功能、日志功能或备份功能时，改动风险和阅读成本都会变高。

建议动作：

- 按功能拆分为最少 3~5 个组件，例如：
  - `SettingsPanel`
  - `StrategyPanel`
  - `BackupPanel`
  - `LogsPanel`

#### 4.4 策略 UX 仍可继续打磨

问题：

- 当前可用，但缺少“保存成功提示 / 预览已过期提示 / 未保存变更提醒 / 风险提示”等细节。

影响：

- 管理员可以完成配置，但在反复调参时容易缺乏操作反馈。

建议动作：

- 增加轻量成功提示
- 标记 preview 是否基于当前未保存配置
- 在 `priority_override` 模式下增加“会丢弃冲突节点”的风险说明

### P2：产品化增强项

#### 4.5 更细粒度策略能力仍未实现

当前未做：

- 按输出目标分别配置策略
- 单上游独立规则
- 单节点例外规则
- 更灵活的去重维度

说明：

- 这些都属于产品化增强，不建议在当前阶段优先于验证闭环和可维护性。

当前文档主干已经基本补齐。

2. 补后端策略接口级测试
3. 拆分 `web/src/App.jsx`
4. 继续做策略产品化增强

下面条件全部满足后，可以认为本轮“策略最小闭环收尾”完成：

- 至少具备一组策略接口级测试
